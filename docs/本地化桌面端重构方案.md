# Night Diary V2 — 本地化桌面端重构方案

> **决策背景**: 放弃 SQL、Redis 等后端数据基础设施，转型为本地优先、像游戏一样双击即用的桌面应用。无需用户登录，所有数据本地存储。
>
> **基于**: V1 架构诊断、V2 task.md 蓝图、cursor rules、README.md 的综合分析。

---

## 一、核心判断：什么变了，什么不变

### 不变的核心资产（从 V1/V2 继承）

| 模块 | 说明 |
|------|------|
| Multi-Agent 架构 | Supervisor + Empathy/Retrieval/Insight Worker，LangGraph StateGraph |
| 意图分类器 | 规则层 + LLM 层两级分类 |
| RAG 检索管线 | Chroma 语义 + BM25 关键词 + RRF 融合 + bge-reranker 精排 |
| Thompson Sampling | Beta 分布强化学习，四种回应风格探索 |
| Skill 系统 | 10 个 Skill + SkillRegistry 贪心选择 |
| 领域知识库 | 85 条心理学知识（CBT/正念/睡眠卫生等），Chroma 向量存储 |
| 三层记忆 | 工作记忆 / 情景记忆 / 长期记忆（实现方式调整） |
| 危机检测 | 极端负面情绪短路 → 安全模板 |
| 执行分级 | light / medium / heavy 三级路由 |
| 前端技术栈 | Vue 3 + TypeScript + Tailwind CSS + Vitest |

### 移除的内容

| 移除项 | 原因 |
|--------|------|
| MySQL 8.0 | 单用户本地应用，SQLite 完全足够 |
| Redis 7 | 单用户单进程，无需分布式缓存/锁/限流 |
| Docker Compose | 目标用户不运行 Docker，应用应双击即用 |
| JWT 认证体系 | 无多用户，无登录 |
| User 模型及相关 | 单用户，无注册/登录/权限 |
| Admin 管理后台 | 无需管理多用户 |
| 公开专栏 | 无社交场景 |
| 限流器 / 分布式锁 | 无并发竞争 |
| CORS 中间件 | 全 localhost 通信 |
| `alembic/` | SQLite 场景可简化为 `create_all()` + 手动迁移脚本 |

## 二、目标架构总览

```
NightDiary.exe (Tauri v2 桌面壳)
│
├─ Rust 主进程
│   ├─ 分配随机空闲端口
│   ├─ 启动 Python 后端子进程（PyInstaller 打包的 .exe）
│   ├─ 健康检查轮询 → 加载 Vue 3 前端
│   └─ 应用退出时优雅关闭 Python 进程
│
├─ Vue 3 前端（Tauri WebView 内运行）
│   ├─ 通过 localhost API 与 Python 通信
│   ├─ 游戏化 UI：粒子背景、玻璃态面板、场景过渡动画
│   └─ 日/夜双主题
│
└─ Python AI 引擎（localhost FastAPI，仅绑定 127.0.0.1）
    ├─ SQLite（日记、分析结果、反馈、标签、配置）
    ├─ ChromaDB（向量存储，本地持久化目录）
    └─ AI 分析管线（Multi-Agent + RAG + Skill）
```

**关键设计决策**：

- **Tauri v2 而非 Electron**：安装包 3-8 MB vs 120+ MB，内存占用 ~50MB vs ~200MB
- **Python 作为 sidecar 进程而非重写为 TypeScript**：LangChain/LangGraph/ChromaDB/sentence-transformers/jieba 等 AI 管线重度依赖 Python 原生生态，重写成本极高
- **PyInstaller 打包 Python 为独立 .exe**：最终用户无需安装 Python
- **SQLite 而非 JSON 文件**：保留 SQLAlchemy ORM 的查询能力和 ACID 保证，仅换数据库 dialect

## 三、数据层设计

### 3.1 数据库：SQLite

SQLAlchemy 2.0 的 dialect 抽象使得切换成本极低——只需修改连接 URL：

```
# 原来
DATABASE_URL = mysql+pymysql://user:pass@localhost:3306/nightdiary

# 现在
DATABASE_URL = sqlite:///{DATA_DIR}/night_diary.db
```

**ORM 模型调整**（从 Phase 1 规划的 10 个模型中精简）：

保留：
- `DiaryEntry` — 日记条目
- `Analysis` — AI 分析结果（含 `execution_tier` 字段）
- `Tag` / `DiaryTag` — 标签
- `ModelProvider` — 用户配置的 LLM
- `Feedback` — 显式/隐式反馈
- `StylePreference` — Thompson Sampling 的 Beta 参数

移除：
- `User` — 无多用户
- `KnowledgeEntry` — 可用 ChromaDB 的 domain_knowledge collection 替代

新增：
- `AppConfig` — key-value 配置表（替代部分 .env 功能）

### 3.2 向量存储：ChromaDB（不变）

ChromaDB 本身就是本地持久化的，无需任何改动。Collection 命名去掉 `user_{uid}_` 前缀：

```
{DATA_DIR}/chroma_data/
├─ diary_chunks        # 所有日记分块的 embedding
└─ domain_knowledge    # 心理学领域知识（只读，首次启动 seed）
```

### 3.3 Redis 功能的替代方案

| 原 Redis 用途 | 替代方案 |
|--------------|---------|
| 情景记忆（Sorted Set） | Python 进程内 `deque[EpisodicEntry]`，max 100 条，LRU 淘汰，可选 SQLite 持久化 |
| 分布式锁 | 删除——单用户单进程 |
| 限流器 | 删除——无 API 滥用面 |
| Token 统计聚合 | SQLite 直接 `SELECT SUM()` |
| 天气缓存 | SQLite 表 + TTL 字段 |

### 3.4 数据目录布局（Windows）

```
%APPDATA%/night-diary/
├── night_diary.db           # SQLite 主库
├── chroma_data/             # ChromaDB 向量库
├── models/                  # 下载的 embedding/reranker 模型
│   ├── text2vec-base-chinese/
│   └── bge-reranker-base/
├── config.json              # 非敏感配置（主题、窗口大小等）
├── secrets.key              # Fernet 密钥（加密 LLM API Key）
├── backups/                 # 自动备份
│   └── 2025-06-01-auto.db
└── logs/
    ├── backend.log
    └── app.log
```

## 四、前后端通信

### 4.1 启动流程

```
用户双击 NightDiary.exe
  → Tauri Rust 进程启动
  → 显示启动画面（Logo + "正在准备 AI 引擎..." + 进度条）
  → 分配随机端口（如 51982）
  → spawn Python 子进程：nightdiary-backend.exe --port 51982 --data-dir %APPDATA%/night-diary
  → 轮询 GET http://127.0.0.1:51982/health（200ms 间隔）
  → 健康检查通过 → 关闭启动画面，加载 Vue 3 WebView
  → Vue 通过 Tauri invoke 获取端口号，所有 API 请求发往 localhost:51982
```

### 4.2 API 端点（精简后）

所有端点都在 `/api/v1/` 下，无需认证中间件：

```
日记：
  GET    /api/v1/diary/entries           # 日记列表（分页）
  POST   /api/v1/diary/entries           # 创建日记
  GET    /api/v1/diary/entries/{id}      # 日记详情
  PUT    /api/v1/diary/entries/{id}      # 更新日记
  DELETE /api/v1/diary/entries/{id}      # 删除日记

AI 分析：
  POST   /api/v1/analysis/{diary_id}     # 触发 AI 分析
  GET    /api/v1/analysis/{diary_id}     # 查看分析结果

反馈：
  POST   /api/v1/feedback/{analysis_id}  # 提交赞/踩

标签：
  GET    /api/v1/tags                    # 标签列表
  POST   /api/v1/tags                    # 创建标签
  DELETE /api/v1/tags/{id}               # 删除标签

LLM 配置：
  GET    /api/v1/models                  # LLM 提供商列表
  POST   /api/v1/models                  # 添加 LLM 配置
  PUT    /api/v1/models/{id}             # 更新 LLM 配置
  DELETE /api/v1/models/{id}             # 删除 LLM 配置

系统：
  GET    /api/v1/stats                   # 统计（日记数、Token 用量等）
  POST   /api/v1/backup                  # 手动备份
  GET    /api/v1/health                  # 健康检查
  POST   /api/v1/shutdown                # 优雅关闭
```

## 六、游戏化 UI 设计要点

目标是让用户感觉"打开了一个陪伴型应用"，而不是"填写网页表单"。

### 6.1 设计原则

| 原则 | 实现 |
|------|------|
| 无浏览器外壳 | `"decorations": false`，自定义标题栏融入整体风格 |
| 深度与质感 | 玻璃态面板（`backdrop-blur` + 半透明 + 柔阴影） |
| 动画优先 | GSAP 编排复杂序列；Vue `<Transition>` 处理路由切换；CSS keyframes 处理环境特效 |
| 环境氛围 | Canvas 粒子系统（日间：暖色浮尘；夜间：星空视差） |
| 音效 | Web Audio API（翻页声、AI 回应铃、书写沙沙声）；默认关闭，可手动开启 |
| 触感反馈 | 按钮 hover 微放大 + 阴影提升；click 微缩小 + 涟漪 |
| 字体 | 正文：Inter/系统无衬线；日记展示：手写体（如 LXGW WenKai） |

### 6.2 场景设计（替代传统路由导航）

不使用导航栏。用"场景"隐喻组织页面：

1. **启动场景** — Logo 动画，AI 引擎启动进度条（能量条样式），粒子背景
2. **首页场景** — "桌面"隐喻：日历贴纸视图、大而醒目的"写日记"按钮（脉冲光晕）、连续天数卡片
3. **日记场景** — 全屏沉浸式：羊皮纸/信纸风格书写区、情绪选择器（emoji 网格，非下拉框）、写完后翻页过渡到 AI 分析
4. **分析场景** — AI 回应以"来信"样式淡入，打字指示器为脉动光点，点赞/踩有满足感的点击动画
5. **回顾场景** — 日历/时间线视图，"书架"隐喻（每个月是一本书）
6. **设置场景** — 干净面板，不是密集表单，分区平滑展开

### 6.3 路由设计

```
/                     → HomeScene（首页）
/write                → DiaryScene（写新日记）
/write/:id            → DiaryScene（编辑日记）
/analysis/:diaryId    → AnalysisScene（查看分析）
/review               → ReviewScene（回顾日历）
/review/:diaryId      → ReviewScene + 条目详情
/settings             → SettingsScene
/settings/llm         → LLM 配置
/settings/backup      → 备份管理
/onboarding           → 首次使用引导（非表单式，叙事式）
```

## 八、AI 模型分发策略

embedding 模型（`text2vec-base-chinese` ~400MB）和 reranker 模型（`bge-reranker-base` ~1GB）体积较大。

**推荐方案：首次启动下载**

- 安装包保持小巧（~200MB，包含 Python 运行时 + 应用代码）
- 启动画面显示"正在准备 AI 引擎..."进度条
- 从 HuggingFace 镜像下载模型到 `%APPDATA%/night-diary/models/`
- 下载完成后写入标记文件，后续启动跳过
- 提供"跳过，稍后下载"选项（此时 AI 分析功能不可用）

## 十、风险和注意事项

1. **PyInstaller 打包体积**：包含 torch + chromadb + sentence-transformers 的 Python 环境打包后约 1.2-1.8GB。这是"游戏级"的体积，在桌面端可接受，但需要做好心理预期。

2. **Tauri v2 的成熟度**：相比 Electron，Tauri v2 较新。如果遇到阻塞性问题，降级备选方案是 Electron（多 ~120MB 但生态成熟）。

3. **ChromaDB 在 PyInstaller 中的兼容性**：ChromaDB 依赖 onnxruntime 等原生 DLL，PyInstaller 打包时需要手动声明 hidden import 和二进制依赖。V1 没有解决过这个问题（V1 是直接 Python 运行），需要额外调试时间。

4. **Windows WebView2 依赖**：Tauri 使用系统 WebView2，Windows 10 1809+ 已内置。如果用户系统缺少，Tauri 安装器可自动下载（约 120MB 一次性依赖）。

5. **embedding 模型首次下载**：国内用户访问 HuggingFace 可能较慢，需要提供镜像源配置（如 hf-mirror.com）或内置模型文件。

6. **数据备份的重要性**：本地 SQLite 单文件虽方便，但也有损坏风险。必须在应用内提供自动备份 + 手动导出功能。
